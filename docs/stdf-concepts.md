# STDF-Inspired Semiconductor Test Data Concepts

## Overview

This document explains the STDF-inspired semiconductor test data layer added to the Semiconductor Test Support Lab.

The purpose of this section is to connect the lab with semiconductor test automation concepts such as ASICs, FPGAs, ATE systems, wafers, test sites, measurement limits, pass/fail results, and yield analysis.

This project does **not** parse real binary STDF files. Instead, it uses structured STDF-inspired CSV data to demonstrate how semiconductor test results can be generated, processed, summarized, validated, and archived through an Application Support Engineering workflow.

## What Is STDF?

STDF stands for **Standard Test Data Format**.

In semiconductor testing, STDF is commonly used to store test result data generated by automated test equipment.

A simplified semiconductor testing flow looks like this:

```text
Device Under Test
   ↓
Automatic Test Equipment
   ↓
Test Program Execution
   ↓
Measured Values
   ↓
PASS / FAIL Results
   ↓
STDF or Test Result Output
   ↓
Analysis / Database / Dashboard / Reports
```

In real environments, STDF files are usually binary files. They may contain detailed information about:

* Test programs
* Test stations
* Lots
* Wafers
* Devices
* Test numbers
* Measurement values
* Limits
* Pass/fail results
* Bin information
* Test execution metadata

## What This Project Does

This lab does not implement a full STDF parser.

Instead, it creates an STDF-inspired dataset using CSV format. This makes the data easier to inspect, process, and document while still representing important semiconductor test concepts.

The project generates records containing fields such as:

```text
timestamp
lot_id
wafer_id
x_coord
y_coord
site_id
tester_id
device_type
device_id
test_number
test_name
measurement_value
low_limit
high_limit
unit
result
fail_reason
```

These fields are used to calculate yield and identify test result patterns by lot, wafer, site, tester, device type, test name, and fail reason.

## ASIC Concept

ASIC stands for **Application-Specific Integrated Circuit**.

An ASIC is a chip designed for a specific purpose. It is optimized for a particular function and is commonly used when performance, power efficiency, or production scale matters.

Examples of ASIC use cases include:

* Networking chips
* Image processing chips
* AI accelerator chips
* Storage controllers
* Custom embedded processors
* Communication chips

In this project, ASIC devices are represented as simulated devices under test.

Example:

```text
device_type = ASIC
device_id = ASIC-001-01-001
```

The lab does not design ASICs. It simulates test result data that could come from devices categorized as ASICs.

## FPGA Concept

FPGA stands for **Field-Programmable Gate Array**.

An FPGA is a reconfigurable chip. Unlike an ASIC, an FPGA can be programmed or reconfigured after manufacturing.

FPGAs are commonly used for:

* Hardware prototyping
* Digital logic testing
* Signal processing
* Accelerators
* Validation environments
* ASIC prototyping

In this project, FPGA devices are also represented as simulated devices under test.

Example:

```text
device_type = FPGA
device_id = FPGA-002-03-004
```

The lab does not program FPGAs. It simulates test result records for FPGA-like devices in a semiconductor testing context.

## ASIC vs FPGA

A simplified comparison:

```text
ASIC = optimized, fixed-purpose chip
FPGA = flexible, reconfigurable chip
```

In an analogy:

```text
ASIC = a Formula 1 car built for one specific race purpose
FPGA = advanced LEGO hardware that can be rebuilt into different designs
```

Both ASICs and FPGAs may need validation, testing, analysis, and support tooling around their test results.

## ATE Concept

ATE stands for **Automatic Test Equipment**.

ATE systems are used to automatically test electronic devices and semiconductor components.

A simplified ATE flow:

```text
ATE system
   ↓
Runs test program
   ↓
Measures electrical or functional behavior
   ↓
Compares measured values against expected limits
   ↓
Generates PASS or FAIL result
   ↓
Stores test data
```

In this project, ATE systems are represented by the `tester_id` field.

Example:

```text
tester_id = ATE-01
tester_id = ATE-02
```

This allows the lab to calculate yield and failures by tester.

## DUT Concept

DUT stands for **Device Under Test**.

The DUT is the device being tested.

In this lab, each simulated device is a DUT.

Example:

```text
device_id = ASIC-001-01-001
device_type = ASIC
```

or:

```text
device_id = FPGA-002-03-004
device_type = FPGA
```

## Lot ID

A lot represents a group of devices manufactured or processed together.

Example:

```text
lot_id = LOT-2026-001
```

In manufacturing, lot-level analysis is important because failures concentrated in one lot may indicate a process or production issue.

Analogy:

```text
A lot is like a batch of cookies made together.
If many cookies from the same batch are bad, the issue may be related to that batch.
```

In this project, yield is calculated by lot to identify whether one lot performs worse than another.

## Wafer ID

A wafer is a thin slice of semiconductor material where many chips are manufactured.

Example:

```text
wafer_id = WAFER-03
```

Wafer-level analysis helps identify whether failures are concentrated on a specific wafer.

Analogy:

```text
A wafer is like a tray full of cookies.
Each cookie represents one chip.
If one tray has more bad cookies than the others, that tray needs investigation.
```

In this project, yield is calculated by wafer.

## X and Y Coordinates

Semiconductor devices on a wafer can be identified by physical coordinates.

Example:

```text
x_coord = 12
y_coord = 8
```

These coordinates represent the position of a device on the wafer.

Analogy:

```text
X and Y coordinates are like row and seat numbers in a theater.
They tell you where the device was located.
```

Although this lab does not generate wafer maps, the coordinates are included to represent realistic semiconductor test data structure.

## Site ID

A test site represents a physical or logical test position on the tester.

ATE systems may test multiple devices in parallel. Each parallel position can be represented as a site.

Example:

```text
site_id = 1
site_id = 2
site_id = 3
site_id = 4
```

Site-level analysis is important because repeated failures on one site may indicate a tester, socket, contact, fixture, or configuration issue.

Analogy:

```text
If four cashiers are serving customers and cashier 3 makes more mistakes than the others, the problem may be cashier 3, not the customers.
```

In this project, site 3 is intentionally given a slightly higher probability of outliers to simulate a possible site-related issue.

## Test Number

A test number is an identifier for a specific test.

Example:

```text
test_number = 1001
test_name = Voltage Test
```

This makes it easier to identify and track tests consistently.

The project includes test numbers such as:

```text
1001 - Voltage Test
1002 - Frequency Test
1003 - Memory BIST
1004 - Leakage Current
1005 - Temperature Test
1006 - Signal Integrity
```

## Test Name

The test name describes what is being validated.

Examples:

```text
Voltage Test
Frequency Test
Memory BIST
Leakage Current
Temperature Test
Signal Integrity
```

Each test has expected measurement limits.

## Measurement Value

The measurement value is the value produced by a test.

Example:

```text
measurement_value = 1.024
unit = V
```

The measured value is compared against low and high limits.

## Low Limit and High Limit

Low and high limits define the acceptable range for a measurement.

Example:

```text
low_limit = 0.95
high_limit = 1.05
measurement_value = 1.024
result = PASS
```

If the measured value is outside the range, the result becomes FAIL.

Example:

```text
low_limit = 0.95
high_limit = 1.05
measurement_value = 1.087
result = FAIL
fail_reason = HIGH_LIMIT
```

## Unit

The unit describes what the measurement represents.

Examples:

```text
V
MHz
%
mA
C
ns
```

Units are important because a measurement value has no clear meaning without its unit.

Example:

```text
1.02 V
```

is not the same as:

```text
1.02 mA
```

## Result

The result field stores whether a test passed or failed.

Example:

```text
result = PASS
```

or:

```text
result = FAIL
```

The result is calculated by comparing the measurement value against the low and high limits.

## Fail Reason

The fail reason explains why a test failed.

Examples:

```text
LOW_LIMIT
HIGH_LIMIT
```

If the test passes, the fail reason is blank in the generated CSV.

In the processed report, blank fail reasons are grouped as:

```text
NONE
```

## Yield

Yield is the percentage of records or devices that passed.

The simplified formula used in this lab is:

```text
Yield = Passed / Total * 100
```

Example:

```text
Total records = 360
Passed = 322
Failed = 38
Yield = 89.44%
```

Yield analysis helps identify how healthy a test run, wafer, lot, site, device type, or test category is.

## Scripts Added

This STDF-inspired layer includes two Python scripts.

### Data Generator

Script:

```text
app/generate_stdf_inspired_data.py
```

Purpose:

```text
Generate structured semiconductor test records inspired by STDF concepts.
```

Output:

```text
data/stdf_inspired_test_records.csv
```

The generator creates records across:

* 2 lots
* 3 wafers
* 10 devices per wafer
* 6 tests per device

Total generated records:

```text
2 lots * 3 wafers * 10 devices * 6 tests = 360 records
```

### Data Processor

Script:

```text
app/process_stdf_inspired_data.py
```

Purpose:

```text
Read the STDF-inspired CSV file and calculate yield metrics and breakdowns.
```

Outputs:

```text
reports/stdf_yield_summary.csv
reports/stdf_yield_breakdown.csv
reports/stdf_yield_report.txt
```
