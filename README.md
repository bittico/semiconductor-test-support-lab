# Semiconductor Test Support Lab

## Overview

This project simulates an Application Support Engineering environment inspired by semiconductor test automation workflows.

The lab demonstrates how an internal engineering support toolchain can execute test simulations, generate reports, store results in a database, validate metrics, publish a dashboard, run health checks, and process STDF-inspired semiconductor test data through a CI/CD pipeline.

The main goal is not to perform real semiconductor testing, but to practice the type of technical workflow an Application Support Engineer may support in an engineering environment.

## Jenkins Pipeline

The Jenkins pipeline automates the full workflow.

Pipeline flow:

```text
GitHub Repository
   ↓
Jenkins Pipeline
   ↓
Jenkinsfile
   ↓
Python Test Simulator
   ↓
CSV Report
   ↓
PostgreSQL Import
   ↓
SQL Metrics Validation
   ↓
Nginx Dashboard Generation
   ↓
STDF-Inspired Data Generation
   ↓
STDF-Inspired Yield Processing
   ↓
Application Health Checks
   ↓
Archived Build Artifacts
```

The pipeline performs the following actions:

* Pulls the project from GitHub.
* Reads the `Jenkinsfile`.
* Verifies that Python is available.
* Runs the semiconductor test simulation script.
* Displays the generated CSV report in the Jenkins console output.
* Imports the CSV report into PostgreSQL.
* Runs SQL metrics validation.
* Generates the Nginx dashboard from PostgreSQL metrics.
* Validates that the dashboard file exists.
* Generates STDF-inspired semiconductor test records.
* Processes STDF-inspired records and calculates yield metrics.
* Validates that STDF-inspired reports exist and contain expected content.
* Runs application health checks.
* Archives the CSV report, dashboard HTML, health check report, and STDF-inspired reports as build artifacts.

More details are available in:

[Jenkins Pipeline Documentation](docs/jenkins-pipeline.md)

[Jenkins PostgreSQL Automation Documentation](docs/jenkins-postgresql-automation.md)

## Jenkins Credentials

The PostgreSQL password is managed using Jenkins Credentials instead of being stored directly in the `Jenkinsfile`.

Credential ID used by the pipeline:

```text
postgres-lab-password
```

This allows Jenkins to inject the password securely during pipeline execution.

In the Jenkins console output, the password is masked:

```text
PGPASSWORD=****
```

This demonstrates a better CI/CD security practice by avoiding plain-text secrets in source control.

## STDF-Inspired Semiconductor Test Data

This project includes an STDF-inspired semiconductor test data layer.

STDF stands for **Standard Test Data Format**. In real semiconductor environments, STDF is commonly used to store test result data generated by automated test equipment.

This project does **not** parse real binary STDF files. Instead, it uses structured CSV data inspired by STDF concepts to demonstrate how semiconductor test data can be generated, processed, summarized, validated, and archived.

More details are available in:

[STDF-Inspired Semiconductor Test Data Concepts](docs/stdf-concepts.md)

## STDF-Inspired Data Generator

The generator script is:

```text
app/generate_stdf_inspired_data.py
```

It generates:

```text
data/stdf_inspired_test_records.csv
```

The generated dataset includes:

* Timestamp
* Lot ID

* Site ID
* Tester ID
* Device type
* Device ID
* Test number
* Test name


## STDF-Inspired Yield Processor

The processor script is:

```text
app/process_stdf_inspired_data.py
```

It reads:

```text

```

and generates:

```text
reports/stdf_yield_summary.csv
reports/stdf_yield_breakdown.csv
reports/stdf_yield_report.txt
```

The processor calculates:

* Total records
* Passed records
* Failed records
* Yield percentage
* Yield by lot
* Yield by wafer
* Yield by site
* Yield by tester
* Yield by device type
* Yield by test name
* Yield by fail reason

This helps identify which test categories, sites, wafers, or device types may require deeper investigation.
